## Session: Active-Only Market Tracking Fix (CRITICAL)

### The Problem
**24 tokens tracked, but only 16% had complete orderbooks.**

Root cause: Tracking past, present, AND future markets:
- Past markets (8 tokens): DEAD - no trading, frozen orderbooks
- Future markets (8 tokens): SPARSE - pre-order bids only
- Current markets (8 tokens): ACTIVE - two-sided orderbooks

When WebSocket sends updates for current market tokens, the binary couldn't find their complements because:
1. Token IDs change each period (new period = new tokens)
2. Complements were in past/future markets (not receiving updates)

### The Fix (From NotebookLM Guidance)

**Changed `src/condition_map.rs`:**
```rust
// BEFORE: [current - 900, current, current + 900] (past, current, future)
fn get_15m_periods() -> Vec<i64> {
    vec![current, current - 900, current + 900]
}

// AFTER: [current, current + 900] (current + next only)
fn get_active_15m_periods() -> Vec<i64> {
    vec![current, current + 900]
}
```

**Also fixed outcomes filter:**
- Before: Only accepted "Yes"/"No" outcomes
- After: Also accepts "Up"/"Down" outcomes (Polymarket BTC/ETH markets use these)

## Technical Details

### Market Lifecycle (NotebookLM)

```
[Pre-order phase] → [Active trading] → [UMA resolution] → [Settled]
   (sparse)           (liquid)          (halted)          (final)
```

- **Pre-order:** Market created, accepting bids, but no tight spreads
- **Active:** Market is live, full liquidity, tight spreads
- **UMA resolution:** Trading halted, awaiting outcome verification
- **Settled:** Final outcome known, positions resolved

### Timestamp Calculations
